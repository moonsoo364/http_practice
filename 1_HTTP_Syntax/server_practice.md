
# 1장 Http Syntax

# 1.4 HTTP의 조상(1) 전자메일
HTTP의 헤더는 요청과 응답 양쪽에서 사용됩니다. 이터넷이 보급되기 전 부터 사용자 간 정보 교환으로 사용되던 메일 시스템에서 이 규칙을 사용했었습니다.
헤더는 클라이언트와 서버 사이에 필요한 추가 정보, 명령 등을 쓰는 장소입니다. 헤더는 종류가 다양한데 아래는 주로 사용하는 헤더를 정리한 것입니다.

## Client Header

### 1. User-Agent

클라이언트가 자신의 애플리케이션 이름을 넣는 곳. curl 커맨드를 사용하면 `curl/7.48.0` 과 같은 문자열이 들어간다. 서버는 이곳의 이름을 보고 응답을 구분하기도 한다. 스마트폰, PC의 경우도 브라우저의 종류에 따라 버전을 구분할 수 있다. 자신의 기능을 포함하는 브라우저의 목록을 나열하는 식으로 이름을 표시한다.

### 2. Referer

서버에서 참고하는 정보 클라이언트가 요청을 보낼 때 보고 있던 페이지의 URL을 보낸다. 페이지의 참조원을 서버가 참조하는데 이용한다.

### 3. Authorization

특별한 클라이언트에만 통신을 허가할 때 인증 정보를 서버에게 전달한다. RFC에서 몇 가지 표준 형식(Basic/Digest/Bearer)를 정했지만 AWS 깃 허브 등에서는 웹 서비스 자체 규격을 요구하기도 한다.

## Server Header

### 1. Content Type

파일 종류를 지정. 여기에는 MIME 타입이라는 식별자를 기술한다. MIME 타입은 전자메일을 위해 만들어진 식별자다.

### 2. Content-Length

바디 크기, 만약 다음 헤더에서 소개하는 압축이 이루어지는 경우 압축 후의 크기가 들어간다.

### 3. Content-Encoding

압축이 이루어진 경우 압축 형식을 설명한다.

### 4. Date

문서가 생성된 날짜이다.

## 1.4.1 헤더의 전송
헤더를 보내기 위해선 `--heder="해더행"` 아니면 단축형인 `-H`을 사용합니다.

curl http 요청 예제 
```powershell
curl.exe --http1.0 http://localhost:18888/greeting
```

헤더 테스트
```powershell
curl.exe --http1.0 -H "X-Test: Hello" http://localhost:18888
```

같은 헤더를 보낼 경우, 서버에 따라 다르다. Go 의 경우 배열로 처리한다.

```powershell
curl.exe --http1.0 -H "X-Test: Hello" -H "X-Test: World" http://localhost:18888
```

자주 사용하는 curl에서 헤더는 alias로 제공한다.
예를 들어 `--user--agent` 는 `-A`를 사용한다.

```powershell
curl.exe --http1.0 -A "Mozilla/5.0 (compatible; MSIE 10.0; Windows NT 6.1; Trident/6.0)" http://localhost:18888/greeting
```

## 1.4.2 헤더 수신
`-v` 옵션을 지정하면 서버에서 반환하는 헤더도 같이 표시됩니다.

```powershell
curl.exe --http1.0 -A "Mozilla/5.0 (compatible; MSIE 10.0; Windows NT 6.1; Trident/6.0)" -v http://localhost:18888/greeting 
```
## 1.4.3 MIME
MIME 타입은 파일의 종류를 구분하는 문자열로 전자메일을 위해 만들어졌습니다.
일반적으로 웹사이트는 html 파일을 바탕으로 사진이나 음악을 붙인 것이 대부분입니다.
파일의 종류에 따라 화면에 표시하거나 저장할 수 있는데 이 때 파일의 종류를 나타내는 식별자가 MIME 타입입니다.

## 1.4.4 Content-Type과 보안
브라우저 세계에서는 Content-Type 헤더에서 지정된 MIME 타입을 사용합니다.

브라우저에서 MIME 타입이 아닌 내용을 보고 파일 형식을 추측하는 것을 컨텐츠 스니핑(Content Sniffing) 이라고하는데
서버 설정이 잘못되어도 사용자 입장에서는 파일이 잘 표시된다. 그러나 원래 텍스트로 표현되야하는 `text/plain` 파일인데도 HTML과 자바스크립트가 있으면
브라우저가 파일을 실행하는 문제가 생겨 보안에 문제가 발생한다.
서버에서는 다음과 같은 헤더를 전송해 브라우저가 파일을 추측하지 못하도록 하는 것이 권장됩니다.

```powershell
X-Content-Type-Options: nosniff
```

# 1.5 HTTP의 조상(2) 뉴스그룹
윈도우 XP 이전에는 뉴스그룹으로 불리는 기사를 읽거나 투고하는 플랫폼이 있었습니다.
뉴스라는 이름이지만 매스미디어가 아닌 대규모 전자 게시판입니다. 
주제별로 그룸이 만들어져, 논의가 이루어지거나 소프트웨어 공개에 사용되곤 했습니다.
파이썬도 'alt.sources' 그룹에 1991년에 투고된 것이 최초의 공개라고 합니다.

뉴스그룹은 분산 아키텍처로 되어 있습니다.
사용자는 서버에 구독하는 최신기사를 요청하고 기사가 있으면 가져옵니다. 웹처럼 모든 사용자가 한곳의 서버에 구독하는 최신 기사를 요청하고 기사가 있으면 가져옵니다.
복수의 서버가 마스터/슬레이브 구조로 연결돼 있어, 슬레이브의 서버도 클라언트처럼 마스터 서버에 접속하고, 정보를 가져와 로컬에 저장합니다. 스토리지 용량에 제한이 있어
모든 기사가 유지되는 것은 아니며 오래된 것부터 지워갑니다. 이 때 클라이언트 - 서버 간 통신에 이용된 것이 NNTP(Network News Transfer Protocol)이라는 규칙입니다.
HTTP/1.0 보다 10년전에 이 규칙이 정의되었으며 HTTP는 뉴스그룹으로부터 메서드와 스테이터스 코드라는 두 가지 기능을 도입했습니다.

## 1.5.1 메서드
HTTP/1.0으로 통신할 때 전송되는 GET 부분은 메서드로 불립니다.
HTTP의 경우 파일 시스템 같은 설계 철학으로 만들어져있습니다. 아래 세가지 메서드가 흔히 쓰이는 메서드입니다.

1. GET: 서버에 헤더와 컨텐츠를 요청
2. HEAD: 서버에 헤더만 요청
3. POST: 새로운 문서 투고

아래 두 메서드는  1.0 이후에도 있어지만 구현에 따라 제공되지 않기도 했던 메서드입니다.
실제 브라우저가 표준 기능만으로 이들 메서드로 서버에 요청을 보낼 수 있게 된 것은 자바스크립트에서 XML. HttpRequest가 지원되고 나서부터 입니다.
HTML 폼에는 GET과 POST만 지원합니다.

1. PUT: 이미 존재하는 URL의 문서를 갱신한다.
2. DELETE: 지정된 URL 문서를 삭제한다. 삭제에 성공하면 URL의 무효화가 된다.

curl 커맨드로 메서드를 전송할때는 `--request=` 혹은 그 단축형인 `-X` 옵션을 사용합니다.

`-X HEAD`의 단축형으로서 `--head`, `-I`  옵션이 있습니다.
Request Body를 전송을 위한 옵션로 `-f`, `-D`가 있는데 이 전송 옵션을 사용하고 `-X` 옵션이 빠진 경우 자동으로 `POST` 메서드로 호출합니다.
```powershell
curl.exe --http1.0 -X POST http://localhost:18888/greeting
```

## 1.5.2 Status Code
스테이터스 코드도 뉴스그룹에서 가져온 기능 중 하나입니다. 이 세 자리 숫자를 보고 서버가 어떻게 응답했는 지 한누에 알 수 있습니다.
크게 5가지 카테고리로 응답을 구분합니다.

1. 100번대: 처리가 계속됨을 나타낸다. 1xx계열은 특수한 용도로 사용된다.
2. 200번대: 성공했을 때의 응답. 자주 사용되는 코드로 200 OK로 정상 종료를 나타낸다.
3. 300번대: 서버에서 클라이언트로의 명령. 오류가 아니라 정상 처리의 범주, 리디렉트나 캐시 이용을 지시한다.
4. 400번대: 클라이언트의 요청에 오류가 있다.
5. 500번대: 서버 내부에서 오류가 발생했다.

# 1.6 리디렉트

300번대 스테이터스의 일부는 서버가 브라우저에 대해 리디렉트하도록 지시하는 스테이터스 코드입니다.
리디렉트에는 5가지 종류가 있습니다.

| Status Code | 메서드 변경 여부 | 영구/일시적 여부 | 캐시 여부 | 설명 |
| --- | --- | --- | --- | --- |
| 301 Moved Permanetly | O | 영구적 | 캐시함 | 도메인 이전, 웹사이트 이전 |
| 302 Found | O | 일시적 | 지시에 따름 | 일시적 관리, 모바일 기반 전송 |
| 303 See Other | O | 일시적 | X | 로그인 후 페이지 전환 |
| 307 Teporary Redirect | X | 일시적 | 지시에 따름 | RFC 7231에서 추가 | 
| 308 Moved Permanetly | X | 영구적 | O | RFC 7231에서 추가 | 

영구적인지 일시적인지는 이전 페이지가 이후에도 존재하는 지 여부로 구분합니다.
예를 들어 새로운 도메인을 얻어 서버의 컨텐츠를 이동한 경우나 HTTP로 운영되던 페이지를 HTTPS로 전환한경우 예전 페이지를 볼일이 없습니다.
이 때는 영구적인 리디렉트가 됩니다.

메서드의 변경은 첫 번째 요청이 POST고 두 번쨰 요청이 이후 GET, HEAD를 사용할 경우 사용자에게 확인없이 메서드 변경을 해도되는 지 여부를 나타낸 것입니다.

현재는 주로 다음과 같이 구분합니다.

1. 301/308: 요청된 페이지를 다른 장소로 이동했을 때 사용된다. 영구적으로 이동하며 검색 엔진도 이 응답을 받으면 기존 페이지의 평가를 새로운 페이지로 계승한다.
2. 302/307: 일시적인 이동이다. 모바일 전용 사이트로 이동하거나 관리 페이지를 표시한다.
3. 303: 요청도니 페이지에 반환할 컨텐츠가 없거나 반환할 페이지가 따로 존재할 경우 그 페이지로 이동시킬 때 사용한다. 예를 들면 로그인한 후 우너래 페이지로 이동하는 경우에 사용한다.

클라이언트는 Location 헤더를 보고 다시 요청합니다.
curl 커멘드에 -L을 부여하면 응답이 300번대고 게다가 응답 헤더에 Location헤더가 있으면 그 헤더에 지정된 URL에 다시 요청을 보냅니다. 기본적으로 최대 50번까지 리디렉트합니다.
리디렉트 횟수도 `--max --redirs` 옵션으로 지정할 수 있습니다.
```powershell
curl.exe --http1.0 -L http://localhost:18888/greeting
```
Go 언어의 기본 설정으로 리디렉트는 10회로 제한되어 있습니다, **308 Moved Permantly**에 따른 리디렉트는 버전 1.8에서 지원하기 시작했습니다.

리디렉트에는 단점이 있습니다. 리디렉트하는 곳이 다른 서버라면 리디렉트할 때마다 TCP 세셔ㅓㄴ 접속, HTTP 송수신으로 두 번 왕복 통신이 발생합니다. 리디렉트 수가 늘어나면
그만큼 표시에 걸리는 시간이 늘어납니다. 구글은 리디렉트 횟수는 5회 이하, 가능하면 3회이하로 가이드라인을 제시 했습니다.

# 1.7 URL
URL은 RFC 1738, 상대 URL은 RFC 1808에서 정의되었습니다.
웹 시스템에서 URL와 URI는 거의 같습니다. RFC 3305에서 URL은 관용 표현으로, 공식표기는 URI가 됐지만 일반적으로 URL이 일반적으로 사용됩니다.
URL은 HTTP를 위해 만들어졌지만 FTP, 뉴스그룹, 메일도 지원하는 규격이 되었습니다.

## 1.7.1 URL의 구조
일반적으로 자주 보는 URL은 다음과 같은 형식입니다.
```shell
https://www.oreilly.co.jp/index.html
```
이 URL은 아래와 같은 요소로 구성됩니다.
**스키마://호스트명/경로**
이 규칙을 따라 요소를 분해하면 각 항목은 다음과 같은 요소가됩니다. 
1. 스키마 : https
2. 호스트 명: www.oreilly.co.jp
3. 경로: index.html

URL 사양에 포함되는 모든 요소가 들어간 예제는 다음과 같은 형식이 됩니다.

**스키마://사용자:패스워드@호스트명:포트/경로#프래그먼트?쿼리**

스키마 해석은 브라우저의 책임입니다. 브라우저는 스키마를 보고 적절한 접속 방법을 선택해야 합니다. 실제로 통신하는 곳은 호스트명으로 지정된 서버입니다.
포트는 아파트 등의 현관 우편함 같은 것입니다. 주소(호스트명으로 찾아온 IP주소)마다 65,535개의 포트가 있습니다. 
같은 서버여도 포트가 다르면 독립적으로 복수의 서버를 운영해 서비스를 제공할 수 있습니다.
포트가 생략되면 스키마별 기본 포트를 사용합니다. HTTP라면 80포트 HTTPS라면 443번 포트 입니다.

사용자 이름이나 패스워드는 FTP에서 사용되며 웹 시스템에는 비밀번호가 누출되는 문제가 있어 사용하지 않습니다.

URL은 주소를 지정하는 데 사용하지만, 동시에 사용자가 읽는 문장이기도 합니다. 예를 들어 다음 URL를 보면
**http://www.oreilly.co.jp/books/9784873114958/**
이 url을 읽은 사람은 우선 호스트 명을 보고 오라일리 재팬 사이트라고 파악합니다. 
경로의 'books'를 보고 서적 관련 페이지라고 알 수 있고 마지막경로인 '974...'을 보고 아마 ISBN코드라고 추측할 수 있습니다.

구글 검색 엔진도 URL에 단어가 포함된 경우에는 그 단어가 검색 순위가 높아지는 효과가 있다고 합니다.
브라우저에 링크가 URL에 표시되었을 때 사람이 읽을 수 있는 URL이 사용자에게 조금이라도 많은 정보를 사전에 맣은 정보를 전달할 수 있습니다.

URL을 구성하는 ASCII 문자열로, 영문자와 숫자 그리고 몇 개의 기호만 그대로 표시할 수 있습니다.
RFC 2718에서는 UTF-8로 UJRL을 인코딩함으로써 다국어 문자도 다룰 수 있게 되었습니다.

URL은 위키피디아의 일본어 항목처럼 확장된 일본어를 포함하거나 쿼리를 갖아 길어지기 쉽습니다.(한자 한 글자가 UTF-8로 3바이트 URL 인코딩으로 9바이트가 됩니다.)
HTTP 사양상 URL 길이에 제한은 없지만 인터넷 익스플로러에서는 2083자까지만 다룰 수 있어 대체로 2000자가 기준이 되었습니다. 
HTTP/2 (RFC 7231)에서는 URL이 지나치게 길 때 반환되는 스테이터스 코드 **414 URI Too Long**이 추가되었습니다. 

# 1.8 바디
HTTP/0.9 사양에서는 요청에 데이터를 포함할 수 없었습니다. 
응답은 파일 컨텐츠 자체였지만, 1.0에서는 요청과 응답 양쪽에 헤더가 포함돼 바디와 헤더를 분리할 필요가 있었습니다.
